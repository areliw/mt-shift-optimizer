# MT Shift Optimizer

เว็บแอปจัดตารางเวร MT (Medical Technologist) — กำหนดบุคลากร กะ และจำนวนวัน แล้วระบบจะสร้างตารางที่กระจายเวรให้เท่ากันและตรงกับ constraint (skill, วันหยุด)

---

## Present ในระบบ — เห็นความเจ๋ง ของระบบได้ชัดเจนที่สุด
**หนึ่งปุ่ม → ได้ตารางเวรทั้งเดือน**  
ไม่ต้องไล่เติมมือ ระบบคิดให้สมดุลและตรงเงื่อนไขทุกข้อ

**ใช้ OR-Tools (Google) ตัวเดียวกับที่ใช้จัดตารางระดับโลก**  
Constraint programming (CP-SAT) — ใส่กฎได้หลากหลาย แล้วให้เครื่องหาคำตอบที่ดีที่สุด

**กฎที่รองรับแบบละเอียด**  
วันหยุดประจำสัปดาห์ + รายเดือน + วันหยุดราชการ (คลิกปฏิทิน) · ช่วงเวลาที่แต่ละคนอยู่ได้ · ทักษะและระดับ · ฉายา (เต็มเวลา/พาร์ทไทม์) · ขั้นต่ำ–สูงสุดกะต่อเดือน · **ห่างกันอย่างน้อย N วัน** (ไม่ติดขอบสัปดาห์/เดือน)

**จับคู่คนได้ตามจริง**  
อยากอยู่ด้วยกัน · ห้ามอยู่เวรเดียวกัน · “คนนี้ทำงานได้เฉพาะเมื่ออยู่เวรเดียวกับอีกคน” — ระบบจัดให้ครบ

**ยืดหยุ่นตามหน่วยงาน**  
หลายกะ (เช้า/บ่าย/ดึก) · กะเฉพาะบางวันหรือรวมวันหยุด · จำกัดสูงสุดต่อสัปดาห์ต่อตำแหน่ง · มีเทมเพลตตัวอย่าง (เช่น Microbiology 18 คน)

**นำเข้า–ส่งออกทั้งระบบ**  
Export/Import JSON  backup หรือย้ายข้อมูลได้ · ดาวน์โหลดตารางเป็น CSV ไปใช้ต่อ

**Deploy ใช้ได้ทันที**  
รันบนเครื่องหรือ push ขึ้น GitHub แล้วให้ Railway deploy — เปิดใช้ผ่าน browser ได้เลย

## ทำอะไรได้ (สรุปฟีเจอร์)

### ตั้งค่าและสร้างตาราง
- **กำหนดช่วงเวลา** — วันเริ่มต้น + จำนวนวัน (หรือเลือก “ทั้งเดือน”)
- **วันหยุดราชการ** — เลือกจากปฏิทิน (คลิกวันที่) ไม่ต้องพิมพ์
- **สร้างตารางเวร** — กดปุ่มเดียว ระบบใช้ **OR-Tools CP-SAT** หาตารางที่สมดุลและตรงเงื่อนไขทั้งหมด

### บุคลากร
- **เพิ่ม/แก้/ลบ** ชื่อ ฉายา (เต็มเวลา/พาร์ทไทม์)
- **วันหยุดประจำสัปดาห์** — เช่น หยุดจันทร์–เสาร์ (0–6)
- **วันหยุดรายเดือน** — ระบุวันที่ของเดือนที่ไม่อยู่
- **ช่วงเวลาที่อยู่ได้** — เช่น 08:00–16:00, 16:00–24:00, 00:00–08:00
- **ทักษะ + ระดับ** — กำหนดทักษะและระดับ (เช่น เจาะเลือด ≥ พอใช้)
- **จำนวนกะต่อเดือน** — ขั้นต่ำ/สูงสุด (min–max ต่อเดือน)
- **ห่างกันอย่างน้อย N วัน** — เช่น 6 วัน = ประมาณไม่เกิน 1 ครั้ง/สัปดาห์ (ไม่ติดขอบสัปดาห์/เดือน)

### กะ (Shift)
- **หลายกะต่อวัน** — เช้า / บ่าย / ดึก แยกกันได้
- **หลายตำแหน่งต่อกะ** — แต่ละตำแหน่งกำหนดช่วงเวลา, จำนวนคน, ฉายาที่อนุญาต (เต็มเวลา/พาร์ทไทม์)
- **กะเฉพาะบางวัน** — เช่น เช้าเฉพาะเสาร์–อาทิตย์ + วันหยุด
- **สูงสุดต่อสัปดาห์** — จำกัดจำนวนครั้งต่อสัปดาห์ต่อตำแหน่ง (หรือไม่จำกัด)

### จับคู่บุคลากร
- **อยู่ด้วยกัน** — พยายามจัดให้ทำงานวันเดียวกัน (soft)
- **ห้ามอยู่ด้วยกัน** — ไม่จัดให้อยู่เวรเดียวกันในวันเดียวกัน (hard)
- **คนที่ 2 ต้องอยู่กับคนที่ 1** — คนที่ 2 ทำงานได้เฉพาะเวรเดียวกับคนที่ 1 (hard)

### นำเข้า/ส่งออก
- **Export JSON** — ดาวน์โหลดข้อมูลทั้งหมด (บุคลากร, กะ, ทักษะ, ฉายา, ช่วงเวลา, คู่บุคลากร)
- **Import JSON** — นำข้อมูลกลับเข้า (แทนที่ของเดิม)
- **Export CSV** — ดาวน์โหลดตารางล่าสุดเป็น CSV

### เทมเพลต
- **Template 2 (Microbiology)** — กะเช้า/บ่าย/ดึก + 18 คน + เงื่อนไขตัวอย่าง (วันหยุด, ฉายา, min_gap_days, คู่ MT2 ต้องอยู่กับ MT1)

### ตัวแก้ (Scheduler)
- เคารพวันหยุดสัปดาห์ + วันหยุดรายเดือน + วันหยุดราชการ
- เคารพช่วงเวลา + ทักษะ + ฉายา (allowed_titles)
- จำกัดจำนวนกะต่อเดือน + ห่างกันอย่างน้อย N วัน
- จำกัดเวรติดกัน (max consecutive days) + ลดการจัดเวรติดกัน (soft)
- จับคู่: อยู่ด้วยกัน / ห้ามอยู่เวรเดียวกัน / คนที่ 2 ต้องอยู่เวรเดียวกับคนที่ 1
- เป้าหมาย: กระจายจำนวนเวรให้เท่ากัน + กระจายตามช่วงเวลา

## Security Environment (สำคัญก่อน deploy)

กำหนดตัวแปรเหล่านี้ใน production:

```bash
API_KEY=<long-random-secret>
ALLOWED_ORIGINS=https://your-app.example.com,https://admin.example.com
SESSION_SECRET=<another-long-random-secret>
DISABLE_WORKSPACE_AUTH=0
```

หมายเหตุ:
- ถ้า environment ถูกมองว่าเป็น production แต่ไม่มี `API_KEY` หรือ `ALLOWED_ORIGINS` หรือ `SESSION_SECRET` แอปจะหยุดทำงานทันที (fail fast)
- `ALLOWED_ORIGINS` ห้ามใช้ `*` ใน production

## Deploy บน Railway

**เชื่อม GitHub (แนะนำ)**  
ใน [Railway Dashboard](https://railway.app/dashboard) → New Project → **Deploy from GitHub repo** → เลือก repo นี้  

เมื่อเชื่อมแล้ว **ทุกครั้งที่ push ขึ้น GitHub** (branch ที่เลือกไว้) Railway จะ build ใหม่และ deploy ให้อัตโนมัติ — ไม่ต้องกดอะไรเพิ่ม แค่ `git push` ก็อัปเดตบน Railway เลย

**แบบใช้ CLI**  
1. ติดตั้ง: `winget install Railway.cli`  
2. `railway login` → `railway init` → `railway up`  

ระบบจะ build จาก `requirements.txt` และรันตาม Procfile (`web: uvicorn main:app --host 0.0.0.0 --port $PORT`)

**หมายเหตุ:** ข้อมูลเก็บใน SQLite บนเครื่องของ Railway — ถ้า redeploy หรือ restart ข้อมูลอาจหาย ถ้าต้องการเก็บถาวรให้เพิ่ม Volume ใน Railway หรือเปลี่ยนไปใช้ DB ภายนอก

**ให้แต่ละ deploy/เครื่องใช้ตารางคนละชุด:** ตั้งตัวแปรสภาพแวดล้อมใน Railway (Variables) อย่างใดอย่างหนึ่ง  
- **`INSTANCE_ID`** — ใส่ค่าคนละอย่างในแต่ละ service (เช่น `หน่วยงานA`, `staging`, `site2`) ระบบจะใช้ไฟล์ DB แยกกัน เช่น `shift_optimizer_หน่วยงานA.db`  
- **`DATABASE_PATH`** — ใส่ path เต็มของไฟล์ DB (เช่น `/data/optimizer.db`) ถ้าต้องการกำหนด path เอง

## วิธีใช้

1. **ตั้งค่า** — กำหนดจำนวนวัน (num_days) แล้วกดบันทึก
2. **บุคลากร** — เพิ่มชื่อ ประเภท (เต็มเวลา/พาร์ทไทม์) skills (donor, xmatch) และวันหยุด (0–6)
3. **กะ** — เพิ่มกะ (เช่น Morning, Afternoon, Night) และจำนวนคนที่ต้องการต่อห้อง (donor, xmatch)
4. **สร้างตาราง** — กดปุ่ม "สร้างตารางเวร" ระบบจะคำนวณและแสดงตารางล่าสุด
5. **Export** — กด "ดาวน์โหลด CSV" เพื่อนำตารางไปใช้ต่อ

## โครงสร้างโปรเจกต์

- `main.py` — FastAPI app, API endpoints, เสิร์ฟ static
- `database.py` — SQLite schema, staff/shift/schedule CRUD
- `scheduler.py` — OR-Tools CP-SAT จัดตาราง
- `config.py` — ข้อมูลเริ่มต้น (ใช้ seed เข้า DB ครั้งแรก)
- `static/` — หน้าเว็บ (index.html, style.css, app.js)
- `shift_optimizer.db` (หรือตาม INSTANCE_ID/DATABASE_PATH) — ฐานข้อมูล (สร้างอัตโนมัติเมื่อรัน)

## API (สรุป)

- `GET/PUT /api/settings/num_days` — จำนวนวัน
- `GET/POST /api/staff` — รายชื่อ staff
- `PUT/DELETE /api/staff/{id}` — แก้/ลบ staff
- `GET/POST /api/shifts` — รายการกะ
- `PUT/DELETE /api/shifts/{id}` — แก้/ลบกะ
- `POST /api/schedule/run` — สร้างตารางใหม่
- `GET /api/schedule/latest` — ตารางล่าสุด
- `GET /api/schedule/export/csv` — ดาวน์โหลด CSV
